# **AVG Unified Task System: Архитектурные Спецификации (Blueprints)**

**Версия:** 2.0 (Расширенная спецификация Agentic Workflow)

**Статус:** Draft / For Review

**Цель:** Детальное техническое руководство для разработки Фазы 2+ (Графовый Оркестратор, Адаптивный LLM Layer, HITL, UX/DX, Телеметрия и Безопасность).

**Концептуальная база:** Данная архитектура отражает современный сдвиг парадигмы от "prompt-chaining" (линейных цепочек вызовов) к "State-Machines" (конечным автоматам) и Agentic Workflows. Это гарантирует предсказуемость системы, возможность аудита каждого шага и гибкость в обработке непредвиденных ошибок генерации.

## **ЧАСТЬ 1\. Архитектура и Ядро (Core Architecture)**

### **📄 Документ A: Архитектура State Graph Оркестратора**

**Суть:** Переход от жестких линейных пайплайнов к графу состояний (State Graph, вдохновлено паттернами LangGraph/XState). Это позволяет агенту безопасно ошибаться, автоматически возвращаться на предыдущие шаги для исправления, запрашивать помощь человека (Human-in-the-loop) и продолжать работу без фатального падения всего пайплайна.

**1\. Схема Состояния (State Schema)**

Состояние графа — это единственный источник истины (Single Source of Truth) для каждого Task Run. На каждом узле граф принимает текущее состояние, мутирует его (или создает новую версию) и передает дальше.

interface TaskState {  
  taskId: string;  
  sessionId: string; // Для связывания цепочки задач в рамках одной сессии  
  messages: BaseMessage\[\]; // Полная история (HumanMessage, AIMessage, ToolMessage, SystemMessage)  
  context: {  
    entities: ExtractedEntity\[\]; // Сущности, выделенные из запроса (даты, имена, концепции)  
    intermediate\_results: Record\<string, any\>; // Временные данные от вызовов API  
    working\_memory: string; // Scratchpad агента для промежуточных размышлений (Chain of Thought)  
  };  
  current\_intent: IntentType | null;  
  selected\_pattern: SolutionPattern | null; // Ссылка на выбранный DAG (Directed Acyclic Graph) решения  
  risks: RiskFactor\[\]; // Найденные галлюцинации, недостаток данных, конфликты политик  
  adequacy\_score: number; // Метрика уверенности 0.0 \- 1.0  
  iteration\_count: number; // Защита от бесконечных циклов (обычно max\_iterations \= 3\)  
  next\_step: string | 'END' | 'SUSPEND'; // Инструкция для роутера графа  
}

**2\. Основные Узлы (Nodes) и их Контракты**

* IntakeNode (Детерминированный):  
  * **Действие:** Очистка ввода (PII scrubbing \- удаление чувствительных данных), определение языка, базовая проверка на безопасность (Safety Guardrails).  
  * **Выход:** Обогащенный объект messages.  
* IntentClassifierNode (LLM-узел):  
  * **Действие:** Анализирует messages и определяет тип задачи. Использует строгий JSON-формат и Few-Shot примеры для повышения точности.  
  * **Выход:** Заполненное поле current\_intent.  
* PatternMatcherNode (Детерминированный):  
  * **Действие:** Подбирает конкретный SolutionPattern (инструкции и набор доступных тулов) на основе интента и доступного контекста.  
* ExecutionNode (Агентный узел):  
  * **Действие:** Запускает специфичный для паттерна подграф (вызов инструментов, RAG, генерация промежуточных артефактов). Может работать в цикле (ReAct \- Reason \+ Act).  
* AdequacyGateNode (LLM-as-a-Judge):  
  * **Действие:** Независимая LLM оценивает результат ExecutionNode по критериям: *Relevance* (релевантность), *Faithfulness* (соответствие фактам из БД), *Helpfulness* (полезность для пользователя).  
  * **Выход:** Обновляет adequacy\_score и risks. Если скор \< 0.7, генерирует фидбек для самоисправления, увеличивает iteration\_count и направляет на RetryEdge.

**3\. Обработка ошибок и отказоустойчивость (Fault Tolerance)**

* **Self-Correction Loop:** Если LLM отдает невалидный JSON (например, забыла закрыть скобку), граф использует RetryEdge с добавлением инструкции: SystemMessage: "Your previous output failed JSON validation with error: {error\_msg}. Fix it.".  
* **Circuit Breaker (Предохранитель):** Если iteration\_count превышает лимит (например, 3 неудачные попытки сгенерировать валидный ответ), срабатывает Circuit Breaker. Выполнение графа прерывается, и пользователю выдается безопасный fallback: "Мне сложно однозначно интерпретировать задачу. Давайте разобьем ее на части."

### **📄 Документ B: Протокол Human-in-the-Loop (HITL) и состояний ожидания**

**Суть:** Агентная система не должна пытаться угадать то, чего не знает (Zero-Hallucination Policy). Она должна уметь "замораживать" свой процесс, делегировать решение человеку (эксперту) и бесшовно возобновлять работу.

**1\. Механика Suspend / Resume (Checkpointing & Hydration)**

* **Suspend (Заморозка):** Когда граф переходит в HumanFallbackNode, он выполняет сериализацию текущего TaskState в распределенную БД (например, Redis для быстрых сессий или PostgreSQL JSONB для долгих). Текущий процесс (Worker) завершается, освобождая вычислительные ресурсы. Бэкенд возвращает клиенту статус 202 Accepted с деталями запроса.  
* **Resume (Возобновление):** Когда пользователь нажимает кнопку или отправляет уточнение через UI, бэкенд извлекает (hydrates) TaskState по taskId, добавляет ответ пользователя как HumanMessage и возобновляет выполнение графа строго с прерванного узла.

**2\. Расширенный Контракт UI-состояний (UI Contract)**

Бэкенд возвращает на клиент типизированный запрос действий, который Frontend рендерит в виде нативных компонентов (не просто текста в чате):

type HITLRequest \=   
  | { type: 'needs\_clarification'; question: string; context\_snippet?: string }  
  | { type: 'guided\_choice'; question: string; options: Array\<{id: string, label: string, implication?: string}\> }  
  | { type: 'needs\_evidence'; entity: string; description: string; upload\_allowed: boolean }  
  | { type: 'action\_approval'; action\_name: string; payload: Record\<string, unknown\>; risk\_level: 'low' | 'high' }; // Запрос на деструктивное действие

*Пример:* guided\_choice может отрендерить две большие карточки в интерфейсе, где описаны последствия (implication) каждого выбора.

**3\. Защита контекста и Управление Жизненным Циклом (TTL)**

Подвешенные сессии имеют состояние SUSPENDED.

* **Hot Storage:** Хранятся в быстром доступе (Redis) в течение 24 часов.  
* **Cold Storage:** После 24 часов, если пользователь не ответил, граф автоматически "просыпается", переходит в узел TimeoutTermination, формирует резюме проделанной работы ("Я собрал данные, но не получил вашего подтверждения...") и сохраняет TaskState в архив (PostgreSQL).

## **ЧАСТЬ 2\. LLM Layer и Инструменты (AI & Skills)**

### **📄 Документ C: Реестр Инструментов (Seed Tools API) и RBAC**

**Суть:**

Ограничение "творчества" LLM строгими детерминированными функциями. Модель не придумывает логику процессов, она использует предоставляемые ей tools (Tool Calling / Function Calling API), действуя как диспетчер.

**1\. Базовые Инструменты (Seed Tools)**

Каждый инструмент сопровождается жесткой JSON Schema и описанием (description), которое LLM использует для понимания, когда этот инструмент применять.

* search\_project\_context(query: string, limit: number): Поиск по векторной базе проекта (RAG). Использует семантический поиск для извлечения релевантных параграфов.  
* extract\_claims(text: string): Выделение проверяемых утверждений из текста пользователя для последующего фактчекинга.  
* query\_sql\_database(sql\_query: string): Выполнение аналитического запроса к Read-Only реплике базы данных.  
* generate\_map\_structure(entities: ExtractedEntity\[\]): Создание JSON-каркaса для отображения на визуальной UI-карте.  
* format\_to\_artifact(content: string, template\_id: string): Строгое форматирование результата без изменения смысла (например, упаковка черновых мыслей в Markdown-отчет).

**2\. Ограничения доступа и Ролевая Модель (RBAC & Tool Permissions)**

Не все инструменты безопасны. Внедряется система разрешений:

* **Safe Tools (Зеленая зона):** search\_project\_context, extract\_claims, format\_to\_artifact. Вызываются моделью автоматически, асинхронно, без ведома пользователя (фоновая работа).  
* **Destructive/Commit Tools (Красная зона):** update\_project\_status, delete\_artifact, send\_email.  
  * **Механика:** Если LLM решает вызвать такой инструмент, граф **всегда** прерывается. Формируется HITLRequest типа action\_approval. Пользователь видит в UI: *"Система хочет обновить статус проекта на 'Завершено'. Разрешить?"*. Инструмент выполняется только после получения криптографически подписанного токена подтверждения от клиента.

### **📄 Документ D: Стратегия Промптинга и Роутинга**

**Суть:**

Управление поведением моделей через температурный режим, строгие системные инструкции и Few-Shot примеры. В концепции Agentic Workflows "Prompt is Code" (Промпт — это программный код).

**1\. Промпт: Intent Classifier (Роутер)**

* **Температура:** T \= 0.0 (минимизация энтропии, максимальная детерминированность).  
* **Формат:** Обязательное использование response\_format: { type: "json\_object" }.  
* **Инструкция (System Prompt):**"Ты — строгий классификатор намерений для корпоративной системы AVG Task System.  
  Твоя единственная задача — проанализировать ввод пользователя и вернуть JSON со строгим полем intent.  
  Допустимые значения intent: develop\_idea, verify\_claim, build\_map, create\_artifact, smalltalk, unknown.  
  ПРИМЕРЫ:  
  User: 'Проверь, сходятся ли цифры в отчете за Q3' \-\> {"intent": "verify\_claim", "confidence": 0.95}  
  User: 'Набросай структуру презентации' \-\> {"intent": "create\_artifact", "confidence": 0.9}  
  User: 'Привет, как дела?' \-\> {"intent": "smalltalk", "confidence": 0.99}  
  Если confidence ниже 0.75 или задача содержит противоречия, выбирай unknown, чтобы система запросила уточнение."

**2\. Промпт: Adequacy Gate (LLM-as-a-Judge)**

* **Температура:** T \= 0.1.  
* **Инструкция (System Prompt):**  
  "Ты — независимый аудитор качества (Quality Gate). Оцени сгенерированный ответ системы (Draft) на соответствие Задаче (Task) и Контексту (Context).  
  Правила:  
  1. Не допускай 'воды' и общих фраз.  
  2. Любое утверждение должно опираться на Контекст. Если данных нет, ответ должен честно об этом говорить.  
  3. Верни JSON: is\_adequate (boolean), score (0-1), risks (массив найденных логических дыр). Если score \< 0.75, подробно опиши почему в поле feedback для запуска механизма самоисправления."

## **ЧАСТЬ 3\. Пользовательский и Административный Опыт (UX/DX)**

### **📄 Документ E: Структура данных Task Run (Telemetry vs UI)**

**Суть:**

Разделение уровней абстракции. Пользователь (User) не должен видеть логи генерации токенов или JSON-схемы. Администратор/Разработчик (Admin) должен иметь возможность провести глубокий дебаг каждого узла графа для оптимизации затрат и качества.

**1\. Модель Admin Trace (Сохраняется в Data Lake/PostgreSQL, доступна в Console)**

Ориентирована на финансовый контроль, аудит безопасности и анализ производительности.

interface AdminTraceEvent {  
  timestamp: string;  
  trace\_id: string;  
  taskId: string; // Связь с глобальным TaskState  
  node\_name: string; // 'IntentClassifierNode', 'AdequacyGateNode'  
  llm\_payload?: {  
    model: string; // 'gpt-4o', 'claude-3.5-sonnet'  
    prompt\_tokens: number;  
    completion\_tokens: number;  
    cost\_usd: number; // Калькуляция стоимости шага  
    latency\_ms: number;  
    raw\_prompt: string; // Полный промпт со вшитым контекстом  
    raw\_response: string;  
    stop\_reason: 'stop' | 'length' | 'tool\_calls';  
  };  
  tool\_execution?: {  
    tool\_name: string;  
    execution\_time\_ms: number;  
    error\_trace?: string;  
  };  
}

**2\. Модель User Timeline (Отправляется на фронтенд для Progress UI)**

Ориентирована на комфорт, прозрачность и доверие. Агрегация тысяч строк технического лога в 3-5 понятных шагов.

interface UserTimelineEvent {  
  id: string;  
  status: 'pending' | 'active' | 'completed' | 'needs\_action' | 'error';  
  label: string; // "Определяю тип задачи", "Ищу доказательства в базе", "Жду вашего решения"  
  description?: string; // "Просмотрено 14 документов"  
  is\_visible: boolean; // Технические шаги (например, самокоррекция LLM) скрыты: is\_visible: false  
}

*UX-Паттерн:* Фронтенд использует Optimistic UI Updates. Переходы между UserTimelineEvent анимируются плавно, чтобы интерфейс не "мигал" при быстрых миллисекундных ответах роутера.

### **📄 Документ F: Контракт Продуктовых Границ (Product Boundary)**

**Суть:**

Жесткая защита от использования дорогой системы не по назначению (превращения в бесплатный ChatGPT для написания стихов или обхода политик безопасности). Четкое управление контекстом на стыке разных интерфейсов (Chat, Canvas, Dashboard).

**1\. Инструкции по отказу (Graceful Degradation & Safety)**

Если IntentClassifier или IntakeNode выявляют выход за границы продукта (Out of Scope, Smalltalk, Jailbreak попытки), LLM-генерация отключается. Возвращается жестко закодированный ответ:

* *Smalltalk:* "Я сфокусирован на решении аналитических и проектных задач. Давайте вернемся к работе: сформулируйте задачу, например, 'Проверь это утверждение' или 'Собери план'."  
* *Safety/Policy Violation:* "Данный запрос нарушает корпоративные политики безопасности. Операция прервана, инцидент залогирован."

**2\. Cross-Surface Управление (Бесшовные переходы между модулями)**

Dialogue (Окно чата) — это только координационный центр (точка входа). Настоящая работа генерируется в Артефактах.

* **Триггер "Map Ready" (Карта готова):** Если в TaskState накоплено \>5 связанных сущностей (entities), система перестает выводить текст в чат. Вместо этого в чате появляется интерактивный виджет \[Открыть рабочую карту\]. При клике экран разделяется, и справа открывается визуальный граф знаний, связанный с текущей сессией.  
* **Триггер "Artifact Ready" (Документ сгенерирован):** Если паттерн create\_artifact завершен, система генерирует полноформатный документ. В чат отправляется только краткое саммари: *"Отчет по рискам Q3 собран"*. Ниже — кнопка \[Открыть в Редакторе\]. Это предотвращает засорение окна диалога "простынями" текста.  
* **Shared Workspace Context:** Чат и Редактор Артефактов синхронизированы. Если пользователь выделяет текст в Редакторе и пишет в чат *"Исправь этот абзац"*, в TaskState автоматически передается выделенный фрагмент как часть context.
