# **Аудит концептуальной архитектуры и плана реализации: AVG Unified Task System**

**Статус документа:** Аудиторский отчет и рекомендации по рефакторингу

**Объект аудита:** Практическая Проектная Концепция AVG Unified Task System (Документы 1 и 2\)

**Роль:** Аудитор современных AI-решений

## **1\. Executive Summary (Краткий вывод)**

Представленная концепция **AVG Unified Task System** демонстрирует высокую степень зрелости и полное соответствие глобальному тренду перехода от "LLM-чатов" к **Agentic Workflows** (агентным рабочим процессам) и **Task-Oriented UI**.

Ключевой постулат *"Пользователь управляет смыслом, AVG управляет процессом"* — это золотой стандарт проектирования AI-продуктов (например, подход Apple Intelligence или корпоративных решений на базе LangGraph/Microsoft AutoGen). Разделение на сырую генерацию (Adaptive LLM Layer) и строгую валидацию (Adequacy Gate) страхует продукт от галлюцинаций и потери доверия пользователей.

**Вердикт:** Концепция фундаментально верна. Однако план реализации (Документ 2\) содержит архитектурные перекосы и фазовые риски. В частности, попытка построить жесткий детерминированный Task Orchestrator (Фаза 4\) до внедрения Adaptive LLM Layer (Фаза 5\) приведет к двойной работе. Функция Adaptive Solution Generator (генерация процессов на лету) несет высочайший риск отказа системы и требует жесткого ограничения (scoping).

## **2\. Глобальный контекст и оценка адекватности**

Как концепция соотносится с мировым опытом 2024-2026 годов:

### **✅ Сильные стороны (Что сделано идеально)**

1. **Guardrails & Validation (Паттерн Защитных Ограждений):** Идея того, что LLM является генератором черновика до AVG validation, а сырой вывод не показывается пользователю — это Enterprise-стандарт. Это решает проблему "AI-уверенности в бреде".  
2. **Progressive Disclosure (Постепенное раскрытие):** Трансляция сложных процессов в простые UI-состояния (Нужно уточнение, Есть риск) вместо логов генерации снижает когнитивную нагрузку.  
3. **Admin Trace vs User Run:** Четкое разделение телеметрии (для админа) и прогресс-бара (для юзера). Это необходимо для отладки AI-агентов.  
4. **Оценка через токены (AI-Assisted Development):** Использование IN/OUT токенов для оценки объема работ говорит о том, что разработка ведется с активным использованием AI-кодеров (Cursor/Copilot). Это современный подход к эстимации.

### **⚠️ Зоны концептуального риска**

1. **Adaptive Solution Generator (Генерация пайплайнов на лету):**  
   * *Проблема:* Заставлять LLM генерировать *код/логику шагов* нового процесса на лету — это огромный риск. Если модель ошибется в логике шага, весь Task Run упадет, вызвав разочарование пользователя.  
   * *Реальность:* Мировой опыт показывает, что LLM лучше справляется с маршрутизацией по *заранее написанным* графам, чем с созданием новых графов графов исполнения в рантайме.  
2. **"Детерминированный RU адаптер" до LLM:**  
   * В Фазе 3 заложен "deterministic RU adapter", а LLM появляется только в Фазе 5\. Создание детерминированного классификатора намерений (Intent Classifier) без LLM в 2026 году — это трата времени на регулярные выражения и жесткие правила (heuristics), которые сломаются при первом сложном запросе пользователя.  
3. **Локализация (I18N) в начале проекта:**  
   * Выделение целой Фазы 2 под мультиязычность интерфейса до того, как собран MVP базового диалога — преждевременная оптимизация.

## **3\. Предложения по архитектурному рефакторингу**

Для снижения рисков предлагаю следующие архитектурные изменения:

### **А. Переход к State Graph (Граф состояний) для Orchestrator**

Вместо монолитного оркестратора используйте архитектуру конечных автоматов на базе графов (вдохновляясь **LangGraph** или **XState**).

Каждый Solution Pattern — это не просто объект, это подграф. Intent Classifier должен быть узлом графа (LLM-based роутером), который перенаправляет задачу на нужный подграф.

### **Б. Ограничение "Adaptive Solution Generator" (Ограниченная свобода)**

Вместо того чтобы позволять системе придумывать *любые* шаги, дайте ей набор детерминированных инструментов (Tools/Skills): "Найти в БД", "Сравнить", "Извлечь сущности", "Сгенерировать документ".

*Адаптивный генератор должен собирать новый процесс **только как лего из существующих, проверенных кубиков**, а не писать новые.*

### **В. Human-in-the-Loop (HITL) как системный примитив**

В концепции упомянуто guided\_choice и needs\_clarification. На архитектурном уровне это должно быть реализовано как пауза исполнения графа (suspend execution). Состояние сохраняется в БД (TaskRunStore), система ждет ввода пользователя, после получения — возобновляет выполнение с прерванного места.

## **4\. Оптимизация плана разработки (Рефакторинг Фаз)**

План требует перегруппировки для более быстрого Time-to-Market (TTM) и проверки самых рискованных гипотез в начале.

### **Текущая проблема плана:**

LLM интегрируется только в Фазе 5\. До этого 4 фазы строится "пустая оболочка", которая не сможет обрабатывать естественный язык.

### **Предлагаемый (Оптимизированный) План:**

| Новая Фаза | Фокус | Обоснование | Изменения относительно оригинала |
| :---- | :---- | :---- | :---- |
| **Phase 1: Product Contract & Architecture** | Проектирование структур данных, контрактов (Task Run, Boundaries). | Фундамент. Без изменений, отличный старт. | (Оригинальная Фаза 1\) |
| **Phase 2: Core Orchestrator \+ LLM Layer (СЛИЯНИЕ)** | Реализация классификатора намерений (Intent) и базового пайплайна **сразу на базе LLM**. | Создавать Intent Classifier без LLM не имеет смысла. LLM нужна оркестратору с первого дня для маршрутизации. | Слияние Фаз 4 и 5\. Сначала подключаем LLM, потом строим вокруг нее роутинг и adequacy gate. |
| **Phase 3: Dialogue MVP & Task Run UI** | Пользовательский интерфейс, "Ход работы", отображение состояний. | Как только есть ядро (Фаза 2), нужно сразу дать возможность с ним взаимодействовать и видеть лог выполнения. | Слияние Фаз 3 и 7\. Юзер и админ должны сразу видеть, как работает ядро. |
| **Phase 4: Seed Solution Library** | Внедрение базовых паттернов (Идея, Проверка, Карта, Артефакт). | Теперь система умеет не просто отвечать, а выполнять сложные многошаговые сценарии. | Оригинальная Фаза 6, но **БЕЗ** Adaptive Generator (откладываем его). |
| **Phase 5: Cross-Surface & Boundaries** | Интеграция с документами, картами, API (POST /dialogue/respond). | Система выходит за рамки чата и начинает порождать артефакты. | Слияние Фаз 8 и 9\. |
| **Phase 6: Advanced Capabilities** | Adaptive Solution Generator, локализация (I18N), расширенная аналитика. | Это "вишенки на торте" и самые рискованные фичи. Их нужно делать, когда ядро стабильно. | Сюда уходит Фаза 2 (Языки) и сложная генерация пайплайнов из Фазы 6\. |
| **Phase 7: E2E QA, Docs & Handoff** | Финальная полировка, проверка краевых случаев (LLM fails). | Без изменений. | (Оригинальная Фаза 10\) |

### **Выгоды от рефакторинга плана:**

1. **Быстрая проверка гипотез (Fail Fast):** Вы протестируете способность LLM адекватно маршрутизировать задачи (Intent Classifier) во 2-й фазе, а не в 5-й.  
2. **Снижение холостых токенов:** Не придется писать детерминированные заглушки в Фазе 3, которые потом будут выброшены в Фазе 5\.  
3. **Ранняя визуализация:** Пользовательский интерфейс "Ход работы" появляется раньше, что критически важно для демо-презентаций стейкхолдерам.

## **5\. Итог**

Ваша концепция описывает передовую Task-Oriented System с отличным пониманием продуктовых границ (Product Boundaries). Если вы примените графовую архитектуру для оркестратора и отложите рискованные фичи (генерацию процессов на лету, мультиязычность) на финальные этапы, сместив интеграцию LLM в самое начало для роутинга, вероятность успешной реализации вырастет кратно. Оценки в IN/OUT токенах выглядят реалистично для команды, использующей AI-инструменты разработки.
