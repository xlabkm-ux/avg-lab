# Specs: Strategic Planning & Roadmap

## Purpose

`.qoder/specs/` contains **strategic plans, product roadmaps, and long-term vision documents** that guide AVG development direction.

While `.qoder/` operating model handles **how** to execute (sprint management, agent coordination, quality gates), specs define **what** to build and **why**.

## Relationship with .qoder/

| Aspect | `.qoder/specs/` | `.qoder/` (operating model) |
|--------|-----------|-----------|
| **Scope** | Strategic (weeks/months) | Tactical (days/sprints) |
| **Content** | Roadmaps, vision, positioning | Protocols, policies, execution |
| **Change Frequency** | Phase-level (weeks) | Task-level (daily) |
| **Primary Agent** | Architect/Product | All agents |
| **Model Tier** | `strong` (analysis) | `standard` (execution) |

## How to Use

### For Architects
1. Read active specs in `specs/`
2. Convert roadmap phases into sprint backlogs in `.qoder/war-room/sprint-backlog.md`
3. Create task cards in `.qoder/war-room/tasks/`
4. Update `specs/integration-map.md` with progress

### For Product Agents
1. Track success metrics in `specs/integration-map.md`
2. Maintain competitive positioning docs
3. Update roadmap based on user feedback

### For All Agents
1. Read relevant specs before sprint execution
2. Understand the "why" behind your tasks
3. Report blockers that affect roadmap timeline

## Model Budget

| Activity | Tier | Agent |
|----------|------|-------|
| Roadmap creation/analysis | `strong` | Architect/Product |
| Sprint conversion | `standard` | Architect |
| Progress tracking | `minimal` | Product |
| Spec updates | `standard` | Architect |

## Core Principle

> "Простыми средствами получить оптимальное и креативное решение"

Strategy through discipline, not complexity.
